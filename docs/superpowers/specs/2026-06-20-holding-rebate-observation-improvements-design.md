## 1. 背景

本轮需求横跨三个模块：

- 持仓分析：统一筛选区密度、修复 sticky 横向滚动透出、统一字段对齐。
- 返费分析：收紧筛选布局、补校验语义、清理无需返费与已全返订单、统一是否可返口径。
- 观察日历：补实时点位 / 当日收盘价、支持存续 / 已完结切换、修正敲出价展示逻辑与页面样式。

这些需求里既有纯样式调整，也有会改变业务口径的后端逻辑。原文档将“已落地项”和“待开发项”混写，执行时容易产生误判，因此这里按“范围 / 状态 / 设计 / 验收”重写。

## 2. 目标

### 2.1 业务目标

- 用户在持仓、返费、观察日历三个页面上看到的字段口径一致。
- 观察日相关的敲出价、点位、派息 / 敲出标签与既定公式一致。
- 返费模块自动区分“待返 / 暂不可返 / 无需返费 / 已全返”。

### 2.2 交互目标

- 宽表横向滚动时，sticky 列和表头不再透出。
- 页面主筛选项尽量一行展示，次要筛选折叠到高级筛选。
- 页面风格保持明亮、简洁、低噪音，减少杂色与无意义强调。

### 2.3 技术目标

- 能复用现有价格缓存、观察记录、返费明细同步逻辑，不重复引入第二套数据通道。
- 文档中的“待实现项”都能落到明确的 API、字段、过滤条件和 UI 表现。

## 4. 统一数据口径

### 4.1 返费数据源

- **最新返款明细**：飞书返费文件夹（`rebateFolderToken`）中日期最新的返款明细工作簿。
- 文件名匹配规则：包含“返款明细”或“返费明细”，且可提取日期。
- 统一按最新日期排序取最后一份。

### 4.2 待返 sheet 的职责

“待返” sheet 是以下字段的唯一数据源：

- 申购费 / 管理费 / 业绩报酬扣税比例
- 管理费实收
- 业绩报酬应收
- 订单是否可返
- 后续将补充解析：未返申购费 / 未返管理费 / 未返业绩报酬

### 4.3 观察日历价格口径

- **存续产品**：`spot_price` 表示“今日实时点位”。
- **已完结产品**：`spot_price` 表示“该观察日当天收盘价”。
- 敲出价统一由业务公式计算，不从前端反推。

## 5. 已完成项设计归档

本节只记录已经落地且后续应保持稳定的方案，避免后续重构时被回退。

### 5.1 首月敲出显示

- `first_knockout_ratio` 后端仍保留原始小数比例，例如 `1.0347`。
- 前端统一使用 `formatRatio2(val)` 展示：
  - `null` / `undefined` -> `--`
  - 其他值 -> `Number(val).toFixed(2)`

涉及页面：

- `ProductAnalysis.vue`
- `CustomerHolding.vue`

### 5.2 宽表 sticky 透出修复

统一策略：

- 宽表使用 `border-collapse: separate; border-spacing: 0`
- sticky 表头 / sticky 首列背景必须不透明
- sticky 单元格 hover 态单独补背景，不能依赖透明叠加
- `z-index` 分层固定：普通单元格 < sticky 列 < sticky 表头

当前落地范围：

- `ProductAnalysis.vue`
- `CustomerHolding.vue`
- `ProductCompletion.vue`
- `RebatePending.vue`
- `RebateCompleted.vue`
- `frontend/assets/main.css` 中 `.data-table`

### 5.3 观察日历页面风格与卡片规则

- 页面整体采用明亮简约风格，白底卡片 + 浅蓝 / 浅绿提示色。
- 观察日卡片内，产品名称独占第一行。
- 后续各业务行按规则展示：
  - 仅敲出：第二行显示 `敲出 + 当日敲出点位`
  - 仅派息：第二行显示 `派息 + 当日派息点位`
  - 同时敲出和派息：分两行显示

### 5.4 敲出价计算规则

统一公式：

```text
敲出价 = 入场价 × 当月敲出比例

当月敲出比例 =
  首个敲出观察日敲出比例 - (存续月份 - 锁定期月份) × 每月递减比例
```

补充规则：

- 当 `存续月份 < 锁定期月份` 时，不观察敲出。
- 这里的“存续月份”取当前观察日所在月份，不取最后一期，也不取前端页面当前月份。
- 观察日历页面显示的是**当期观察日对应的敲出价**，不是最终一期敲出价。

## 6. 待实现项详细设计

### 6.1 驯鹿48号数据订正

### 目标

确认“鹿秀驯鹿48号”在 2026-06-22 的敲出价与标的价格为何偏离预期，结论必须基于数据排查，不允许先改口径再解释。

### 排查路径

1. 查询产品主数据：
   - `issue_date`
   - `lock_months`
   - `first_knockout_ratio`
   - `monthly_decrease`
   - `code`
2. 查询该产品 2026-06-22 的观察记录：
   - `months_since_entry`
   - `knockout_price`
   - `underlying_price`
3. 查询价格源映射：
   - 恒科 ETF 是否走 `513180.SH`
   - `prices/client.go` 对应 `secid` 与单位缩放是否正确

### 候选根因

- `issue_date` 错误，导致 `months_since_entry` 计算偏大。
- 恒科 ETF 价格源解析错误，导致返回值偏离。
- 观察记录生成时未取到当日价格，`underlying_price` 为空或旧值。

### 处理原则

- 如果是数据错：改数据并重新生成观察记录。
- 如果是代码错：修复价格源或月份计算，再重新生成观察记录。
- 在根因没确认前，文档与代码都不能声明“已修复”。

### 6.2 待返费分析页：校验列

### 目标

在待返费表中补一组只读“校验”列，用于比对系统计算的未返金额与返款明细里的未返金额是否一致。

### 表头设计

- 位置：`是否可返` 之后、`本次拟返` 之前
- 分组表头：`校验`
- 子列：
  - `申购费`
  - `管理费`
  - `业绩报酬`

### 后端设计

在 `rebatePending` 响应中新增：

- `check_subscribe`
- `check_management`
- `check_performance`

取值规则：

- 比对本系统当前计算的 `outstanding_*`
- 与最新返款明细“待返” sheet 中同订单、同费用的未返金额比较
- 差值绝对值 `<= 0.01` -> `"T"`
- 否则 -> `"F"`
- 若 sheet 中缺列或订单未命中 -> `"--"`

### 前端设计

- `T` 用绿色胶囊或文本
- `F` 用红色胶囊或文本
- `--` 用灰色弱化展示
- CSV 导出追加三列校验结果

### 依赖

必须先明确“待返” sheet 中这四类列位：

- 订单号
- 未返申购费
- 未返管理费
- 未返业绩报酬

未确认列位前，前端表头可先保留，值统一输出 `--`。

### 6.3 无返费订单过滤

### 规则

若订单以下三项返还比例全部为 `0`、空值或不可解析：

- `subscribe_fee_ratio`
- `management_fee_ratio`
- `performance_fee_ratio`

则该订单不属于返费分析范围，不应出现在待返费列表。

### 落地点

- 优先放在后端查询或聚合层过滤
- 不建议只在前端隐藏，否则导出与统计会失真

### 6.4 扣税比例搜不到显示 "-"

### 规则

- 扣税比例数据仍然来源于最新返款明细“待返” sheet。
- 若该订单某项费用在 sheet 中找不到扣税比例：
  - 页面显示 `-`
  - 不再显示“暂不可返”

### 技术约束

- “展示为空”与“业务上是否可返”是两个概念，不能混用。
- `fmtPct` 等展示函数要把历史遗留的“暂不可返 / 不可返”等占位统一转换为 `-`。

### 6.5 是否可返自动判定

### 业务口径

`是否可返` 改为只读业务状态，不再允许页面手工点选。

判定规则：

- 有返还对象 + 至少一项返还比例有效 + 在待返 sheet 中命中 -> `待返`
- 有返还对象 + 至少一项返还比例有效 + 在待返 sheet 中未命中 -> `暂不可返`

补充排除：

- 若订单属于 `待返`，且三项未返金额都 `<= 0`，则视为已全返，不出现在待返费列表。
- 若订单属于 `暂不可返`，即便当前未返金额为 `0`，也保留展示。

### 前端变化

- 筛选下拉由 `全部 / 是 / 否` 改为 `全部 / 待返 / 暂不可返`
- 表格列由可点击按钮改为只读标签
- 删除或废弃 `toggleReturnable` 的手动写接口

### 后端变化

- `rebatePending` 在返回前统一计算 `is_returnable`
- 同时做两类过滤：
  - 三比例全空 / 全 0 的订单直接过滤
  - `待返` 且三项未返金额均 `<= 0` 的订单过滤

## 7. 观察日历接口设计

> 该部分是后续维护基线。若现有代码继续演进，应保持字段含义不变。

### 7.1 接口

`GET /api/observations/calendar?month=YYYY-MM&status=ongoing|completed`

- `status` 缺省为 `ongoing`

### 7.2 返回字段

`CalendarProduct` 新增：

- `spot_price *float64`

含义：

- `ongoing`：今日实时点位
- `completed`：该观察日当天收盘价

### 7.3 存续产品取价

- 先读本地缓存：`PriceByDate(code, today)`
- 缓存缺失时，批量调 `prices.FetchAll`
- 成功后 `UpsertPrice`
- 避免页面每次打开都逐个请求外部行情接口

### 7.4 已完结产品取价

- 查询已完结产品集合
- 观察日仍由 `observations.DatesForMonth` 计算
- 当日点位优先取观察记录中的 `underlying_price`
- 无观察记录时返回 `nil`

## 8. UI 与样式约束

### 8.1 表格统一规范

- 两行表头：
  - 第一行分组表头居中
  - 第二行子表头按页面需要设置，但默认靠左
- 所有数据单元格统一左对齐
- 数值列保留 `tabular-nums`，但不强制右对齐

### 8.2 筛选区规范

- 主筛选项优先放第一行
- 次要筛选项放高级筛选折叠区
- 输入框宽度使用收紧版 token，避免无意义留白
- 查询 / 重置按钮在大屏贴右，小屏自动换行到下一行右侧

### 8.3 页面风格规范

- 明亮、低饱和、少色彩混杂
- 主要强调色使用品牌蓝，辅助状态色只保留绿 / 红 / 黄
- 不使用高饱和紫色、深色大面积底色、重阴影
